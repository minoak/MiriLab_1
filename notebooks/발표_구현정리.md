# 미리랩 — 구현 정리 (발표용 읽기 문서)

> **미리랩 = 정책을 배포하기 *전에*, AI 가상 시민 사회에서 *미리* 반응을 실험하는 랩(Lab).**
>
> 이 문서는 "무엇을 만들었나"보다 **"왜 이 모양이 되었나"** 를 따라간다.
> 미리랩의 진짜 이야기는 기능 개수가 아니라, **야심찬 8기능 비전을 어떻게·왜 좁혔고,
> 어떤 막다른 길(시행착오)을 거쳐 지금의 단단한 골격에 도달했는가**이다.

---

## 1. 문제의식 — 정책은 왜 시민에게 닿지 못하는가

정부·지자체는 매년 수천 건의 복지·지원 정책을 발표한다. 그런데 **정작 필요한 시민에게 전달되지 못하는** 구조적 누수가 반복된다.

| 사각지대 유형 | 내용 |
|---|---|
| **정보 격차** | 시민이 자신이 정책 대상자인지조차 모른다 |
| **이해도 부족** | 정책 문구가 어렵고 신청 조건이 불명확하다 |
| **디지털 장벽** | 시니어·정보 취약계층은 온라인 신청 자체가 벽이다 |
| **도달 실패** | 발표돼도 필요한 사람에게 닿지 못한다 ("그림의 떡") |

**기존 서비스의 한계** — 정부24 챗봇·복지로는 "정책을 *찾는* 사람"을 돕는다(검색·매칭). 그러나 **"정책을 *만드는* 사람"(입안자)** 이 "이 정책, 배포하면 누가 못 받을까?"를 미리 보는 도구는 없다.

> **미리랩의 한 줄 포지셔닝:**
> *"정책을 시민 입장에서 미리 실험해, 누가 이해 못 하고 · 누가 신청 못 하고 · 어디서 막히는지를 배포 전에 보여준다."*

## 2. 스코프의 진화 — 우리가 무엇을, 왜 덜어냈나 ★

미리랩의 설계는 한 번에 나오지 않았다. **세 번의 큰 방향 전환**을 거쳤고, 그 전환의 "왜"가 이 프로젝트의 핵심 주장이다.

```
[출발: 개요의 8기능 비전]
 RAG · 시민Agent · SNS채팅 · 감성대시보드 · 전파그래프
 · 게시판 자동답변 · AB테스트 · 쉬운글 변환
        │
        │  ① "전파 그래프"가 흔들린다
        ▼
[1차 전환]  전파 그래프(diffusion) 폐기 → "미리 마을"(시간 경과 영향)
        │
        │  ② "24명 마을"이 무겁고 검증 공격에 약하다
        ▼
[2차 전환]  24명 마을 → 대조 3명 "정책 인생극장" (DESIGN v3 확정)
        │
        │  ③ "예측"이라 부르면 검증 공격을 받는다
        ▼
[정직한 포지셔닝]  "예측"이 아니라 "영향 시나리오 / 정책 인생극장"
```

### 2-1. 1차 전환 — 전파 그래프를 버린 이유

개요의 핵심 볼거리 중 하나가 "복지관 → 시니어 → 자녀 단톡방 → 지역 커뮤니티"로 정보가 퍼지는 **전파 네트워크 그래프**였다. 그런데 구현하며 근거가 흔들렸다:

> **우리 페르소나는 실제 한국 인구통계에서 *무작위 샘플링*한 서로 모르는 독립 개인들이다.**
> 이들 사이에 "정보가 전파되는 엣지"를 그리는 건 **근거 없는 연출**이다.

→ 전파(네트워크)를 버리고, **"정책이 한 개인의 삶 궤적에 시점별로 미치는 차등 영향 + 사각지대"** 로 축을 옮겼다. `graph/diffusion.py` 삭제.

### 2-2. 2차 전환 — 24명 마을 → 대조 3명 (DESIGN v3)

"미리 마을"을 24명으로 돌려봤다. 두 가지 문제:

- **범위 과다** — 3~5일 해커톤에 24명 × 여러 시점 시뮬은 무겁다.
- **검증 공격에 약함** — "24명이 대표성이 있냐?"는 질문에 방어가 어렵다.

→ **대조적인 3명**으로 확정. *수혜 / 경계 / 사각지대* 세 사람의 인생이 같은 정책 아래 어떻게 갈리는지를 한 화면에 담는다.

| 후보 | 문제 | 판정 |
|---|---|---|
| 1명 | "인생이 *갈린다*"는 핵심 메시지의 *갈림*이 안 보임 | ✗ |
| 24명 마을 | 범위 과다 + 대표성 검증 공격 | ✗ |
| **대조 3명** | "수혜/경계/사각"이 한 화면에, 구현은 가볍게 | **✓ 스윗스팟** |

> **핵심 장면(발표용):** 같은 *청년 월세* 정책, 6개월 후 —
> A(26세 청년)는 첫 저축을 시작 / B(46세 중장년)는 자녀 위해 대신 신청 / C(74세 고령)는 정책 존재조차 모름.
> **"같은 정책, 다른 인생."**

### 2-3. 정직한 포지셔닝 — "예측"이라 부르지 않는다

미리랩은 미래를 **맞히는** 도구가 아니다. 그렇게 주장하는 순간 "그게 맞느냐"는 검증 공격을 받는다. 그래서 일관되게 **"영향 시나리오 / 정책 인생극장"** 으로 포지셔닝하고, 신뢰성은 **예측 정확도가 아니라 ablation(아래 5장)** 으로 방어한다.

### 3-1. 페르소나 파이프라인 — "손으로 안 만든다"

개요 초안은 "시민 페르소나 20~30명을 *직접 설계*"였다. 우리는 이를 거부하고 **실제 통계 기반 데이터셋**을 택했다.

- **출처:** `nvidia/Nemotron-Personas-Korea` (통계청·대법원·건보공단 등 기반 합성, 약 100만 행 × 26컬럼, CC BY 4.0)
- **다운로드 전략:** 전체는 ~2GB(9 샤드). **첫 parquet 샤드 1개(~220MB)만** 받아 `df.sample(random_state=seed)` — 재현성 + 용량 절약.
- **캐시 우선:** 같은 `(n, seed)`면 네트워크 없이 `personas_cache.json` 재사용 → 오프라인·발표 안정성.
- **결정론 signals 파생:** 데이터셋에 *없는* 3개 신호를 직업·나이·학력에서 **순수 파이썬(LLM 0)** 으로 유도 — 같은 입력 → 같은 출력.

| 신호 | 유도 방식 |
|---|---|
| `digital_literacy` (0~1) | 나이 기본점수 × 학력 배율 × 직업 키워드 가감 (예: 65세↑ 급락, 'IT/사무'↑, '무직/농림'↓) |
| `income_level` (low/mid/high) | 직업 키워드 우선(무직·학생→low, 전문직·관리자→high) + 학력 보정 |
| `government_trust` (0~1) | uuid의 md5 해시 기반 지터 `0.5 ± 0.10` (같은 사람 → 항상 같은 값) |
| `social_network` (list) | 가구형태·직업 기반 태그(독거→복지관·경로당, 자녀→가족 단톡방 …) |

### 3-2. 축 A — 시민 반응 (react → interact → aggregate)

LangGraph 3노드 파이프라인. OpenAI `gpt-4o-mini` 구조화 출력(pydantic):

```
react      : 시민 각자가 정책 1차 반응 + 5축 점수
   ↓         (이해도 / 수혜가능성 / 신청의향 / 불만도 / 공유가능성, 0~100)
interact   : 다른 시민 반응 요약(digest)을 보고 2차 상호작용(입장 변화)
   ↓
aggregate  : 분석가가 전체 집계 → 갈등/합의 요약 + 수정안(개선 반영 정책) + 개선안
```

**산출:** 시민반응 **히트맵**(전원 × 5축, valence 색) + **게이지 3종**(정책수용도 / 신청의향지수 / 사회혼란도).

### 3-3. 축 B — 정책 인생극장 (DESIGN v3 본체)

```
정책 패키지
  └─ 타깃 명세(spec) 추출        : 나이·소득·가구 조건
  └─ 전원 시간경과 시뮬           : 주민 × 시점(1·3·6개월)마다 LLM 1회
  │     place(닿은 채널) + reached_via(경로) + action(서사) +
  │     policy_status + barrier(막힌 지점) + economic/wellbeing + note
  └─ 누적 → 인물별 timeline(인생 궤적)
  └─ 실제 결과에서 대조 3명 선별   : 수혜 / 경계 / 사각지대
```

**채널(장소)이 사각지대의 핵심 표현장치**다(`graph/spaces.py`, 5곳: 복지로 온라인 / 주민센터 / 복지관 / 직장·시장 / 집). 청년은 복지로 5분, 고령은 어디에도 못 닿아 `home`에 갇힘 = 사각지대.

### 3-4. 축과 분리된 별도 무대 — 미리마을 (Generative Agents 오버월드) ★

축 A·B는 정책의 *반응/영향*을 분석한다. 그러나 Park et al.의 원형 — **에이전트가 공간에서 자기 스케줄대로 살아가고 서로 마주치는 "살아있는 마을"** — 은 거기 없었다. **미리마을**은 그 골격을 **완전히 독립된 브라우저 무대**(포켓몬식 오버월드, `미리마을/index.html`)로 구현한 1차 결과다.

> **왜 별도 무대인가:** 메인 Streamlit 앱(state.py 계약·LangGraph)을 *건드리지 않는 섬*으로 분리했다. 실험적 골격이 본체를 깨지 않게. (`graph.llm` 의존도 끊어 미리마을 폴더만으로 독립 실행)

**1차 범위 = planning.** 정책은 아직 없이, 손으로 만든 **10명 고정 캐스트**가 *각자 LLM이 짠 하루 일과(시간→장소→행동)대로* 마을에서 사는지부터 확인한다. (논문 5요소 — memory·retrieval·reflection·planning·interaction — 중 **planning**을 먼저.)

핵심 설계 결정:

| 결정 | 내용 / 왜 |
|---|---|
| **LLM이 하루를 생성** | 손 시간표 단계를 건너뛰고 1차부터 페르소나·관계 기반으로 LLM이 하루 계획 |
| **"녹화 후 재생" 구조** | 백엔드(`gen_schedules.py`)가 하루 전체를 1회 생성→`schedules.json`, 브라우저는 **LLM 0콜로 재생**(즉시·반복·오프라인 = 발표 안전망). ← GA 원본도 백엔드 시뮬→프론트 replay |
| **프롬프트 캐싱 3층** | 고정 prefix(마을·캐스트 명단·정체성·관계) → 가변(요청). "고정→가변" 순서로 OpenAI 자동 prefix 캐싱을 노림 → 2차 멀티턴 비용 대비 |
| **waypoint 그래프 + BFS** | 맵 도로점 652개를 28px 병합해 연결 그래프로 → 어느 건물이든 길찾기(왕복 루프를 목표지향 라우팅으로 대체) |

> **결과 — 관계 창발 ★:** 정체성과 *관계*를 prefix에 넣자, **시키지 않았는데** LLM이 캐릭터들을 서로의 일정에 엮었다 — 민수·수아가 분수광장에서 산책, 영희·미영이 카페에서 점심, 영희가 퇴근 후 김할머니에게 안부 전화. Generative Agents의 **사회적 emergence**가 작게나마 나타난 순간.

### 3-4-2. 살아있는 마을로 — 좌표 정밀화 + 맥락 대화 (A단계) ★

1차가 "10명이 일과대로 *움직인다*"였다면, 그 위에 두 가지를 얹어 마을을 *살아있게* 했다: 캐릭터가 **자기 집·자리에 정확히 서고**, 마주치면 **그날 동선이 묻어나는 대화**를 한다.

#### (1) 좌표 편집기 — "추정하지 말고, 직접 찍게 한다"

- **증상:** 10명이 한 점에서 시작해 뭉치고, 활동 장소에 도착해도 건물 밖 도로에 떴다.
- **원인:** 집이 **단일 노드 하나**(`houses`)였다 — 전원의 home 좌표가 같다.
- **결정:** 멀티모달로 좌표를 *추정*하는 대신, **사용자가 맵을 클릭해 직접 찍는 편집기**를 줬다(픽셀 오차 0 + 통제권). 집 8채 개별화(가족은 공유) + 장소별 "건물 안 도착 박스" → `anchors.js`에 영구 저장. **시작 분포 폭 60px→480px**로 뭉침 해소.
  - *덤으로 잡은 정합 버그:* 스프라이트는 발이 좌표(`-90%`)인데 편집 박스는 중심(`-50%`)이라 ~22px 어긋남 → 박스를 발 기준으로 보정(WYSIWYG).

#### (2) 맥락 대화 — "녹화 틀어주기"에서 "그날 서사"로

- **증상:** 마주치면 쌍별 고정 대사("안녕하세요!")만 반복. *맥락이 없다* — 아침에 만나든 밤에 만나든 같은 말.
- **결정:** 대화를 **그 시점 맥락(각자 그때까지 한 일 + 관계)으로 LLM이 생성**. 여전히 **녹화**(하루치 미리 생성→0콜 재생 = 발표 안전망).
- **재생 UX(사용자 지정):** 만남 발생 → **시계 정지** + 말풍선 → *말풍선 클릭 = 대사창 / 다른 곳 클릭 = 흘려보냄.* 게임 NPC 대화의 그 감각.

```
[ 미리마을 대화 파이프라인 — 전부 녹화(미리 생성 → 재생) ]

 gen_schedules.py  ─ 하루 일과를 LLM 이 생성 (시간 → 장소 → 행동)
       │  schedules.json / village_data.js
       ▼
 gen_dialogues.py  ─ ① 도착 시뮬로 '실제로 만날 수 있는' 만남만 추출
       │            ② 만남마다 그 시점 맥락 → LLM 이 맥락 대화 생성
       │  meetings.js
       ▼
 index.html        ─ 브라우저가 재생: 맵·좌표·도보 이동 → 만남 → 정지 → 대사창
                     (LLM 0콜 · 즉시·반복·오프라인)
```

#### (3) ★ 핵심 시행착오 — "스케줄은 순간이동, 재생은 도보"

- **증상:** 스케줄 겹침으로 만남 14개를 뽑았는데, 막상 재생하니 **절반(7개)이 안 떴다.**
- **원인:** LLM이 짠 스케줄은 *"08:00 집 → 08:00 버스정류장"* 처럼 **순간이동을 가정**한다. 그러나 재생은 캐릭터가 **실제로 걸어간다.** 집에서 먼 곳에 30분만 머무는 만남은 **도착하기도 전에 끝난다.**
- **결정:** 만남을 "스케줄상 겹침"이 아니라 **"실제 걸어 도착하는 시간을 시뮬해, 진짜 함께 머물 수 있는 구간"** 으로만 추출(도로 우회 근사 `ROUTE_FACTOR`). → 만남 6개, **발화 6/6 보장**(못 만난 건 화면 로그로 가시화). *재생과 생성이 같은 이동 모델을 쓰게 맞춘 것.*

> **결과 — 맥락 창발 ★:** 같은 박사장이 카페에서 세 사람과 **각각 다른** 대화를 한다 —
> 다은 *"오전에 텃밭 가꾸고 다시 카페로 왔어요"* · 민수 *"주민센터 세미나 다녀와 구직에 도움될 것 같아요"* · 수아 *"진로 고민이…"*.
> 그날 각자의 동선·관심사가 대화에 묻어난다. **1차의 '관계 창발'에 이어 '맥락 창발'.**

**다음(B단계) = 누적 기억.** 하루를 시간순으로 굴리며 각 캐릭터가 *만남·대화를 기억*하고, 다음 대화가 그 기억을 이어받게(아까 한 얘기를 나중에 언급). A의 만남·대화 파이프라인에 '기억 stream'만 얹는다. *(좌표 미세조정·만남 수 늘리기·행동 타임라인 뷰는 다듬기 백로그.)*

## 4. 핵심 설계 결정 & 시행착오 ★ (본문)

여기가 미리랩의 진짜 내용이다. 각 항목은 **무엇이 틀렸고(증상) → 왜 그랬고(원인) → 어떻게 고쳤나(결정)** 순서다.

### 4-1. 대조 3명 선별: "수치 *예측*" → "실제 *결과* 기반" — 두 심판 문제

- **증상:** 대조 3명 라벨(수혜/경계/사각)이 카드 *서사*와 어긋났다. 신민재가 '경계'로 뽑혔는데, LLM이 쓴 이야기는 "월세를 *받았다*"였다.
- **원인 = 두 심판 문제:** 라벨을 정하는 심판(직업→소득 매트릭스, *시뮬 전* 점수)과 이야기를 쓰는 심판(LLM, *시뮬 후* 서사)이 **달랐다.** 두 심판은 당연히 어긋난다.
- **핵심 통찰:** *수혜/경계/사각은 점수가 아니라 **결과(받음/막힘/못 닿음)** 다.* 반응 점수(benefit/intent)로 뽑아도 같은 함정 — 임수빈은 자기 수혜성을 18점으로 매기지만, 실제로는 자격이 되는데 부모 동거 요건에 막힌 **사각지대**다.

| | Before (예측) | After (결과 기반) |
|---|---|---|
| 라벨 근거 | 시뮬 *전* 매트릭스 점수 | 시뮬 *후* 실제 궤적 결과 |
| 심판 | 둘(점수 vs 서사) | **하나**(같은 궤적에서 라벨·이야기 둘 다) |
| 결과 | 라벨 ≠ 서사 (모순) | 라벨 = 서사 (구조적으로 일치) |

> **결정:** **전원을 먼저 '살게' 한 뒤**, 실제 결과에서 3명을 고른다(`select_trio_from_outcomes`). 24명 전원 시뮬 비용은 OK로 확인됨. 라벨과 이야기가 *같은 데서* 나오므로 어긋날 수 없다.
> *(기존 매트릭스 선별 `select_contrast_trio`는 삭제하지 않고 dormant 안전망으로 남김.)*

### 4-2. 비대상 오라벨 → "태그 대상 게이트" (사실 vs 예측 분리)

- **증상:** 박성인(46세)이 ***청년* 자산형성** 정책에 '사각지대 / 자격은 되는데 막힘'으로 오라벨됐다. 46세는 청년이 아닌데.
- **원인 2겹:** ① 상태 가드가 `aware→unaware`(그냥 관심 끊음)를 'blocked'로 잘못 승격. ② "blocked면 자격은 됨"이라는 가정이 *비대상*에게도 적용됨.
- **결정 — 원칙을 둘로 쪼갬:**
  > **"대상 여부 = 태그(나이·소득·가구)로 판정하는 *사실*. 결과(수혜/막힘) = LLM 궤적."**

  `is_target(persona, specs)`: 선언된 대상 범위를 **AND 하드 체크**(나이가 결정적). 비대상은 결과와 무관하게 '대상 아님(무관)'으로 빠지고 3장 카드 후보에서 제외된다.
  - *추가로 잡은 버그:* mock의 `income_level`은 한글('중간소득'), spec은 영문('mid')이라 정규화 없이는 전원 탈락 → `_norm_income` 한↔영 정규화로 해결.

### 4-3. '사회혼란도' 게이지의 3번 변신 — 이름과 측정을 일치시키기

세 번째 게이지를 무엇으로 둘지가 가장 길게 논의됐다. **이름이 측정과 일치하는가**가 기준이었다.

```
v1. 사회혼란도 = 찬반 양극화           →  ✗ "전쟁=다 같이 반대"인데 혼란도가 낮게 나옴
                                          (이름 ≠ 측정) + stance 분포 바와 중복
v2. 사각지대 = mean(benefit − intent)  →  ✗ 데모 mock에선 benefit≈intent라 게이지 ≈ 0
                                          (특수 공식이 게이지엔 무겁고, 인생극장 사각과 개념 충돌)
v3. 사회혼란도 = mean(dissatisfaction)  →  ✓ '불만의 평균' = '사회적으로 시끄러워질 세기'
   (= 반발 강도)                            이름 = 측정. 데모서도 ~43으로 살아남.
```

> **결정:** "입안자가 가장 두려운 건 *이거 터지지 않을까(반발)*"라는 통찰에서, **반발의 세기 = 불만 점수 평균**으로 단순화(`metrics_common.social_unrest`). 세 경로(real/demo/fallback)에서 같은 공식을 쓰도록 한 곳에 모음.
> *찬반이 갈리는 양상*은 게이지가 아니라 시민반응 탭의 **stance 분포 바**가 직접 보여주고, *왜* 화났는지(형평성·박탈감)는 **페르소나 서사**가 맡는다 — 정량은 게이지, 정성은 서사로 분리.

### 4-4. 프롬프트 현실성 원칙 — "10억 지급" 버그와 과교정

- **증상:** 테스트로 "전 국민에게 10억 지급"을 넣었더니 **전원 찬성**, 혼란도 21. 시민이 '이해득실 계산기'(내가 받나/신청하나)만 보고 실현 가능성·재원·부작용을 무시 — *일차원적*.
- **결정:** `build_react_messages`의 system 첫머리에 **"■ 가장 중요한 원칙 — 현실적으로 판단할 것"** 을 전면 배치. 재원·물가·증세·형평성 부작용·과대공약 의심을 본능적으로 함께 따지되, *합리적 정책엔 냉소 금지*라는 균형 장치 포함.
- **검증:** 10억 지급 → **전원 반대로 전환**(현실성 작동 ✓). **그러나** 정상 정책(청년 월세)도 혼란도가 과하게 오르는 **과교정 신호** 발견 — 모든 정책에 baseline 우려가 깔림.
- **정직한 노트:** 과교정 미세 튜닝(우려를 '무리한 정도'에 비례)은 **팀(프롬프트·UI 담당)에 위임**으로 남김. 시행착오를 *숨기지 않고 기록*한 사례.

### 4-5. 상태 비퇴행 가드 — LLM이 "신청 → 모름"으로 퇴행하는 버그

- **증상:** 실제 API 출력에서 조유정(경계)이 `신청(applied) → 모름(unaware)`으로 **퇴행**. 서류에 막혀 포기한 걸 LLM이 'unaware'로 잘못 표기 → 여정·결론이 모순.
- **결정 — 프롬프트 + 코드 가드 이중화:**
  - *프롬프트:* "상태는 되돌아가지 않는다. 시도했다 포기 = blocked. unaware는 *끝까지 몰랐을 때만*."
  - *코드 가드(`_guard_status`):* LLM이 확률적이라 코드로 보증. `applied→unaware`면 **blocked로 자동 교정**(+막힌 사유 채움). 단, `aware→unaware`(그냥 관심 끊음)는 *blocked로 단정하지 않고* aware 유지 — 4-2의 비대상 오라벨을 막는 핵심.
- **결과:** 조유정이 오히려 "자격은 되는데 서류에 막힘 = 도움 필요"의 **최고의 사각지대 사례**가 됨.

### 4-6. 접근 '방향' 추적 — reached_via / barrier 구조화

- **사용자 통찰:** "사람들이 정책에 *어떤 방향으로* 접근하고, *어디서* 막혔는지 추적이 안 보인다."
- **선택:** 가벼운 길(프롬프트만)과 무거운 길(스키마 구조화) 중 **구조화** 선택. `VillageStep`에 `reached_via`(경로·계기)와 `barrier`(막힌 지점)를 **additive** 필드로 추가.
- **표현:** 펼친 카드 = 🧭 여정 띠(`알게됨→신청→⛔막힘`) + 사건별 산문 + 결론 콜아웃. *"딸이 대신 알려줌"* 같은 풍부한 개인 경로는 실모드 LLM이 채우고, mock은 채널별 일반 템플릿.
- **데이터는 이미 있었다:** 사건(place·policy_status)은 진작 timeline에 있었고, **화면이 안 보여준 것**뿐 — 시뮬·비용 추가 0으로 가시성만 끌어올림.

## 5. 신뢰성 검증 — ablation (grounding ON / OFF)

미리랩은 "예측 정확도"로 신뢰성을 주장하지 않는다(2-3장). 대신 **Park et al. 스타일 컴포넌트 ablation**으로 방어한다.

```
grounded = True   : 페르소나(나이·소득·디지털·신뢰)를 프롬프트에 주입 (실험군)
grounded = False  : 배경 제거, 평범한 익명 시민 가정     (대조군)
```

> **주장:** "**grounding이 이 차이를 만든다.**" grounding을 끄면 세 인물의 궤적이 비슷하게 뭉개지는지(인물 간 분산 ON ≫ OFF)를 정량 비교한다. RAG가 아니라 *페르소나 grounding 자체*가 검증 대상이라, RAG 없이도 신뢰성 논증이 성립한다.

## 7. 한계와 정직한 노트

발표에서 *숨기지 않고 먼저 밝힐* 것들:

- **소득 판정은 근사다.** `income_level`은 직업→버킷 휴리스틱이라 나이만큼 확실하지 않다(나이는 하드 체크).
- **과교정 미해결.** 현실성 프롬프트가 정상 정책의 혼란도도 다소 올린다 → 팀 튜닝 위임.
- **데모 풀이 작다.** 실모드 캐시는 8~24명. "대표성"은 인생극장이 *예측이 아니라 시나리오*라는 포지셔닝으로 방어.
- **3명 대조의 대표성.** 의도된 좁힘. 위에 *전체 풀 분포 헤드라인*(수혜/막힘/자격되나미수혜/대상아님 인원수)을 얹어 "3명은 깊은 서사, 분포는 대표성 숫자"로 두 렌즈 분리.

## 8. 확장 비전 — 슬라이드의 미래 그림 (구현 범위 밖)

좁힌 것은 *버린 게 아니라* 확장 경로로 남겼다:

- **RAG 슬라이스** — 정책 태그(이미 "미래 RAG 검색 키"로 설계됨) + 페르소나 텍스트 7종 인덱싱. 데이터 스키마는 footer-only로 미리 받아둠(`data/dataset_schema.json`).
- **미리마을 (Generative Agents 오버월드)** — "3명 인생극장 → 공간에서 *스케줄대로 사는* 마을"로 확장. **1차(planning) + A단계(좌표 정밀화·맥락 대화) 구현 완료**(3-4장: 10명이 LLM 일과대로 생활하고, 마주치면 그날 동선이 묻어나는 대화 — 관계 창발에 이어 맥락 창발). 다음 = B(누적 기억) → 실시간(시점별) + 정책 주입.
- **개요의 나머지 기능** — SNS 채팅 애니메이션 / 게시판 자동답변 / AB 테스트 심화 (탭 골격은 8개 모두 존재).
