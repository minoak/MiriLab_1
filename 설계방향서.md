# 미리랩 설계 방향서 (재설계 기준) v1.2

> 시스템 수정의 토대 문서. "무엇을 왜 이렇게 바꾸는가"의 단일 기준.
> 발표 정리와 별개 — 이건 *구현 청사진*이다.
> v1.1 (2026-06-06): v1을 코드 대조 검토 후 합의 수정 — acting 한 점 경계 → 상태 사다리 + 전이 가드,
> 축3 사후 확정 → 읽기 전용, 캐시 2층 구조. 변경 사유는 §1.5 표 참조.
> v1.2 (2026-06-06): 지표 소유권 통일 — 게이지 포함 **모든 지표 계산을 축3로 이관**,
> 대시보드는 '정책 결과 종합' 1단으로. 변경 사유는 §1.6 표 참조.

---

## 0. 한 줄 원칙

**판단·서사는 LLM, 전이 규칙·집계는 코드. 축1이 상태를 만들고, 축2가 시간 속에서 굴리고, 축3는 읽기만 한다.**

결과를 새로 판정하는 단계는 없다 — 축2의 시간 전개가 가드 아래 만든 종점이 곧 결과이고,
축3는 그것을 세어(코드) 서술할(LLM) 뿐이다.

---

## 1. 무엇이 문제였나 (재설계 동기 — v1과 동일)

기존 구조는 **두 축이 대등하게 병존**하며 *같은 것(신청·수령)을 다른 해상도로 두 번* 다뤘다.

- 축1(react)이 한 호출에 인지+감정+의향+행동+결과예측을 전부 생성 → 과밀.
- 축2(village)도 행동 서사와 결과(`policy_status`)를 동시에 생성 → 축1과 어긋날 표면 존재.
- 결과: 같은 사람이 두 화면에서 모순될 수 있음 (예: 축1 `no_need` ↔ 축2 `received`).

진단: **책임의 중복.** 단, v1의 해소책(t0 acting 게이트 + 축3 사후 무효화)은 시간을 죽이는
부작용이 있었다 — 그 수정이 v1.1이다.

## 1.5 v1 → v1.1 변경점

| v1 | v1.1 | 왜 |
|---|---|---|
| 경계 = t0 acting 한 점 ("신청을 결정해야만 수령") | 경계 = **사다리** ("사다리를 밟아야만 수령") | t0 고정은 늦깎이 수혜·전파·'시도하면 막힘' 발견(임수빈 류)을 구조적으로 차단 — 시간축 시뮬의 존재 이유와 충돌 |
| 축3 코드가 결과 확정, acting 없으면 received 사후 무효화 | 결과는 축2 종점(가드 통제). 축3는 읽기·집계만, 사후 재라벨 금지 | 사후 무효화는 서사("받았다")와 라벨의 불일치를 새로 생산 — 잡으려던 모순의 재생산 |
| 축1 사다리에 unaware 포함 | 축1은 강제 노출 인터뷰 — 도달(unaware)은 축2 소관 | 축1에 두면 도달 예측이 두 축에서 이중으로 일어남 — §1이 비판한 중복의 재발 |
| 축3 결과 3범주 (received/blocked/unaware) | 사다리 종점 5범주 (현행 contrast 코드와 호환) | '알고도 멈춤'을 unaware에 넣으면 처방(홍보 강화)이 오진 — §4의 "처방이 다르다" 논리의 일관 적용 |
| 캐싱 = 세 축 공유 고정 prefix | 캐시 2층: 결과 체크포인트(축 경계, 이틀) + prefix(호출 내부, 발표 후) | 축간 prefix 공유는 system 상이로 불가. 진짜 절감은 호출 수를 줄이는 체크포인트. react·설문 프롬프트는 갭 실험 측정 도구라 발표 전 변경 금지 |
| 낙차 = 신청의향지수 vs 수령률 | 낙차 = 같은 모수(대상자)·같은 단위(인원 비율) | 의향지수는 점수 블렌드(전원 모수), 수령률은 인원 비율(대상자 모수) — 단위·모수가 달라 뺄 수 없는 두 수였음 |

## 1.6 v1.1 → v1.2 변경점 (지표 소유권 통일)

| v1.1 | v1.2 | 왜 |
|---|---|---|
| t0 지표(수용도·혼란도·의향지수)는 축1(그래프 aggregate)이 계산해 게이지로 먼저 표시, 결과 지표만 축3 | **모든 지표는 축3가 계산** — t0 지표(전원 모수) + 결과 지표(대상자 모수)를 한 자리에서 | "집계는 축3"(§0)의 일관 적용. 축1은 반응·행동을 *결정*하고, 축2는 그대로 *시뮬*하고, 축3가 *정리*한다 — 축1이 먼저 지표를 만들어 두는 건 책임 중복(§1과 같은 병). 실무적으로도 지표 공식이 3경로(ui/mock·graph/nodes·ui/model)로 흩어져 키 불일치·demo/real 공식 분리 버그의 뿌리였음 |
| '축1 지표 선표시'(단계적 표시) | 폐기 — 전 지표를 축3에서 일괄 | 현 구현은 어차피 스피너 안에서 세 축을 완주한 뒤 화면이 뜸 — 선표시는 문서에만 있고 실익 0. 필요해지면 축3를 t0만으로 부분 호출하는 rerun 2단 구조로 재도입(축3가 village=None 을 허용하므로 가능) |
| 대시보드 = 핵심지표(축1 게이지) → 정책 결과(축3) → 신청 여정 분석(t0 추정) 3단 병렬 | **'정책 결과 종합(축3)' 1단** + 신청 여정 분석은 '사전 가설' expander 로 강등 | 실측 깔때기(축3)와 추정 퍼널(t0 점수)이 같은 시각물(같은 빌더 함수)로 위아래 병존 — 같은 질문("어디서 새나")에 두 답. §1이 비판한 '같은 것을 두 번'의 시각화 판 |

## 2. 목표 구조 — 단방향 3축 파이프라인

```
축1  정보 (LLM)
     강제 노출 인터뷰 → t0 메타정보(의향·인지·성향) = 불변 기록
     "알게 되면 신청할 사람인가"
            │  시딩(출발점) — 축2는 이 기록을 수정하지 않고 그 뒤에 시간을 쌓는다
            ▼
축2  결과 (LLM 제안 + 코드 가드)
     행동 사다리의 시간 전개: unaware → aware → applied → received / blocked
     가드: 비퇴행(기존) + 다리 가드(신규: applied 이상엔 reached_via,
           blocked 엔 barrier 필수 — 없으면 1회 재생성)
     시계열 = 변화의 기록, 종점 = 결과
            │  읽기
            ▼
축3  요약·개선 (코드 집계 + LLM 서술)
     코드: t0 지표(수용도·의향지수·혼란도 — 전원 모수)
           + t0 × 시계열 → 수령률·사각지대·낙차·전향·깔때기 (세기만, 판단 0)
     LLM: 확정된 수치의 서술 — 요약·개선안·카드 내러티브
```

**불변 규칙 3개:**
1. **단방향 = 역류 금지.** 축2는 축1의 t0 기록을 수정하지 않는다(시계열 누적만).
   t0 보존이 낙차 측정과 전향 추적의 전제다.
2. **전이 권한은 가드에.** "모르는데 받음"은 다리 가드가 차단 — 받았다면 닿은 경로가
   기록되어야 하므로 구조적으로 불가능. 전향(t0 의향 없음 → 설득 → 수령)은 다리가 있으면
   합법 — 모순이 아니라 *추적된 변화*다.
3. **축3는 읽기만.** 코드는 세고, LLM은 문장만. 라벨 뒤집기(사후 무효화) 금지 —
   잔존 위반은 정직 노트("모순 감지 n건")로 노출한다.

## 3. 축별 계약 (입력 / 책임 / 출력 / 금지)

### 축1 — 정보 (LLM)
- **입력**: 정책(강제 노출) + 페르소나 시트
- **책임**: 정책을 접했을 때의 t0 반응 — 이해·감정·입장·신청 성향
- **출력**: 이틀 = 기존 Reaction 스키마 그대로 (intent·stance·behavior 를 '성향'으로 의미
  재정의). 발표 후 = 의향 사다리(관심 없음 → 관심 → 신청 마음)로 교체 검토.
  unaware 는 축1 출력이 아니다 — 도달은 축2 소관.
- **❌ 금지**: 도달 예측, 신청 성공/수령 예측, 화면 지표 소유(v1.2 — 모든 지표는 축3.
  그래프 aggregate 의 숫자 집계는 하위 호환으로 남되 화면이 읽지 않는다).

### 축2 — 결과 (LLM + 가드)
- **입력**: 축1 시딩(t0 성향) + 정책·채널 환경
- **책임**: LLM 이 스텝마다 전이·서사를 제안하고, 코드 가드가 규칙을 강제한다.
  - 비퇴행 가드(기존 `_guard_status`): 알게 된 사실은 없던 일이 되지 않는다.
  - 다리 가드(신규): applied/received/blocked 엔 reached_via, blocked 엔 barrier 필수.
    위반 시 그 주민만 1회 재생성, 잔존은 교정 + 정직 노트.
  - 시딩 정합: 프롬프트의 1차 반응 블록을 소프트 연속("자연스럽게 이어짐")에서
    하드 출발점("t0 성향에서 출발, 변화에는 계기가 필요")으로 격상.
- **출력**: 시계열 (VillageStep 그대로 — place/reached_via/action/policy_status/barrier/
  economic/wellbeing). 종점 = 결과.
- **❌ 금지**: 다리 없는 전이, t0 시딩 무시.

### 축3 — 요약·개선 (코드 + LLM)
- **입력**: t0 기록(축1) + 시계열(축2) — 읽기 전용
- **책임(코드)**: t0 지표(수용도·신청의향지수·사회혼란도 — t0 기록만 사용, 전원 모수)
  + 결과 범주(§4) · 수령률 · 사각지대율 · 낙차 · 전향 수 · 깔때기 집계(대상자 모수).
  화면의 모든 지표 숫자는 여기서만 나온다(단일 진실원 — 게이지 포함, v1.2).
  화면은 각 숫자에 모수를 명시한다(시민 n / 대상자 n).
- **책임(LLM)**: 확정된 수치의 서술 — 요약 · 개선안 · 카드 내러티브.
- **❌ 금지**: 새 판정, 숫자·라벨 변경.

## 4. 경계 — 사다리와 결과 범주

```
축1 t0 성향 ──시딩──▶ 축2 사다리: unaware → aware → applied → received
                                            │            └→ blocked
                                            └ (멈춤 가능: 알고도 안 감)

축3 결과 범주 (모수 = 대상자 is_target, 비대상 out 은 분포·표에만):
  received      ✅ 수혜          (+ 전향 플래그: t0 의향 없음 → 수령)
  blocked       ⛔ 사각 — 집행 실패  → 채널·절차 처방
  unaware       🚫 사각 — 전파 실패  → 홍보·도달 처방
  aware 멈춤    ⚠ 알고도 안 감     → 진입 장벽·신뢰 처방 (사각과 별도 표기)
  applied 대기  ⏳ 진행 중
사각지대율 = blocked + unaware
```

핵심 효과: received 앞에 닿은 경로(다리)가 강제된다 → "모르는데 받음" 구조적 불가능.
전향은 서사가 있는 합법 전이로 살아남고, 전향 플래그로 추적된다.

**사각 처방의 분화 (v1 §4 계승 + 1범주 추가):**
- `blocked` = 신청까지 갔으나 환경이 막음 → 집행 수정
- `unaware` = 끝내 닿지 못함 → 전파 수정
- `aware 멈춤` = 알지만 안 움직임 → 진입(절차 부담·신뢰·효능감) 수정

## 5. 핵심 지표 — 같은 모수, 같은 단위

- 모수: **대상자(is_target)**. 단위: **인원 비율**. (갭 실험에서 확립한 "같은 단계·같은 단위" 규율)
- **낙차 = t0 의향 적극 비율(intent≥60) − 종점 수령 비율**
  - 양수: 의향이 어딘가서 샜다 → 깔때기로 위치 추적
  - 음수: 전파가 의향을 끌어올렸다 → 그 자체로 발견
- **깔때기(상세 뷰)**: 의향 → 도달(aware 이상) → 신청(applied 이상) → 수령.
  갭별 처방이 다르다: 홍보 / 진입 / 심사.
- (v1.2) **모든 지표는 축3가 센다.** t0 지표(수용도·혼란도·의향지수 — t0 기록만 사용,
  전원 모수) + 결과 지표(수령률·사각·낙차 — 대상자 모수)를 '정책 결과 종합' 한 자리에서
  모수를 명시해 표시. 공식은 현행 실모드 것을 metrics_common 으로 옮길 뿐 **재정의하지
  않는다** — mock 자체 공식은 폐기(demo/real 공식 통일).
- **비신청형 정책**(만 나이 통일·종부세류 — 신청·수령이 없음): policy_spec
  (support_type/channel)으로 분기 — 낙차 헤드라인을 숨기고 t0 지표 중심 표시.

## 6. 실행 흐름 — 캐시 2층

```
실행 버튼 1회:
  축1 전수 react ──(체크포인트)──▶ 축2 전원×스텝 ──(체크포인트)──▶ 축3 집계·서술
```

- **층1 — 결과 체크포인트(축 경계, 이틀)**: 중간 산출물을 session_state(+파일 캐시본)에
  보존. 재실행은 바뀐 축부터만 —
  - 인생극장 재실행 = 축2부터 (축1의 전수 호출 통째 절약)
  - 표시·집계 옵션 변경 = 축3만 (코드라 공짜)
  - 정책 수정·A/B = 축1부터
  호출 수 자체를 줄이므로 토큰 할인보다 절감폭이 크다. 발표 데모 = mock/실런 캐시본.
- **층2 — prefix 캐시(호출 내부, 발표 후)**: 이틀엔 graph/llm.py 에 cached_tokens
  로깅만(무해). 레이아웃 변경(고정 블록 선두 규칙)은 발표 후 —
  - react·설문 프롬프트는 갭 실험 측정 도구(변경 시 리베이스라인 필요)
  - 현 레이아웃(인물카드 선두)이 스텝 간·A/B 간 캐시에 이미 유리할 수 있음 — 실측 후 결정
  - 축간 공유는 불가(system 부터 다름). 축내/호출 패턴별 공유만.

## 9. 한 문단 요약 (시스템 수정 시 항상 이걸로 돌아온다)

미리랩은 **축1(정보) → 축2(결과) → 축3(요약·개선)의 단방향 파이프라인**이다.
축1은 강제 노출 인터뷰로 t0 성향을 만들어 불변 기록으로 남기고,
축2는 그것을 시딩 삼아 행동 사다리를 시간 속에서 굴리되 전이는 코드 가드(비퇴행+다리)가 강제하며,
축3는 t0와 시계열을 읽어 세고(코드) 서술한다(LLM) — 새 판정이 없으므로 축1·2와 어긋날 수 없다.
경계는 사다리다: **사다리를 밟아야만 수령이 가능**해 "모르는데 받음"은 구조적으로 불가능하고,
전향은 모순이 아니라 추적된 변화다. 핵심 지표는 같은 모수·같은 단위의
**t0 의향 ↔ 종점 수령 낙차**이며, 그 깔때기 분해(의향→도달→신청→수령)가 사각의 위치와 처방을 드러낸다.
그리고 게이지를 포함한 **모든 지표 숫자는 축3 한 곳에서만 나온다**(v1.2) —
축1·2는 기록을 만들고, 세는 곳은 하나다.
